## Webserver Enumeration Methodology

### Passive Recon

1. Look at public DNS records for domains/subdomains to enumerate
2. Look at certificates and other public info for domains/subdomains to enumerate

### Active Recon

1. Add domain to `/etc/hosts` file (if applicable)

2. Run directory/page brute force discovery
- [ ] First fuzz for extension (if available)
- [ ] If extension is discovered, fuzz for directories and other pages (with extension appended)

3. Look in `robots.txt` / `sitemap.xml` for hidden endpoints
- [ ] `https://www.example.com/robots.txt`
- [ ] `https://www.example.com/sitemap.xml`

4. Run subdomain/vhost brute force discovery (RUN MULTIPLE WORDLISTS)
- [ ] Run subdomain brute force discovery (External Only)

```bash
ffuf -u https://FUZZ.example.com -w $wordlist -v
```

- [ ] Run vhosts brute force discovery (Internal & External)

```bash
ffuf -u http://example.com -H "Host: FUZZ.example.com" -w $wordlist -fc 301,302,404
```

5. Crawl the webpage for all links
- [ ] Crawling with katana

```bash
katana -u http://example.com -d 3 -o crawl.txt
```

- [ ] Crawling with FinalRecon

```bash
finalrecon --crawl http://example.com
```

- [ ] Crawling with ZAP Proxy

6. Look for comments in HTML for sensitive information
- [ ] Check all crawled and discovered pages

7. Capture server errors (e.g. 500, 403) that might leak tech stack info

8. Look for vulnerabilities in Web Server technologies being used
- [ ] Use Wappalyzer to discover web server technologies in browser
- [ ] Use BuiltWith to discover web server technologies (External only)
- [ ] Use WhatWeb CLI tool to discover web server technologies

```bash
whatweb http://example.com
whatweb -a 3 http://example.com  # aggressive mode
```

- [ ] Scan the webserver with Nikto

```bash
nikto -h http://example.com
nikto -h http://example.com -p 8080 -ssl
```

- [ ] Scan the webserver with NMAP and web discovery scripts

```bash
nmap -sV --script=http-enum,http-headers,http-methods,http-title -p 80,443 $rhost
```

- [ ] Attempt banner grabbing with curl

```bash
curl -I http://example.com
curl -v http://example.com 2>&1 | grep -i "server\|x-powered"
```

- [ ] Discover WAFs with Wafw00f

```bash
wafw00f http://example.com
wafw00f -a http://example.com  # test all WAFs
```

9. If the webserver is running NodeJS or MongoDB, look for NoSQL injection vulnerabilities

10. Look for web service versions on discovered pages (Jenkins, WP, blog platforms, etc.)
- [ ] Look for CMS or app-specific files (`wp-content`, `.git/`, etc.)
- [ ] WordPress Enumeration [[7.0.Web-Application-Analysis]] (CMS Enumeration)
- [ ] Joomla Enumeration [[7.0.Web-Application-Analysis]] (CMS Enumeration)
- [ ] Drupal Enumeration [[7.0.Web-Application-Analysis]] (CMS Enumeration)
- [ ] Tomcat Enumeration (Need to add to [[7.0.Web-Application-Analysis]])
- [ ] Jenkins Enumeration (Need to add to [[7.0.Web-Application-Analysis]])
- [ ] IIS Tilde Enumeration (Need to add to [[7.0.Web-Application-Analysis]])

11. Look for vulnerabilities in discovered web service versions and/or technologies
- [ ] Search Metasploit, ExploitDB, and Google for `<Service> <Version> Exploit GitHub`

12. Look for login pages to test default or weak credentials
- [ ] Default Credential lists
- [ ] Login Brute Forcing
- [ ] Check for password reset or recovery mechanisms
- [ ] Review cookies & sessions (flags like Secure, HttpOnly)
- [ ] Test for session fixation or missing CSRF protections

13. Test submitting data on EVERY user input field and look at interaction with Burp Suite
- [ ] Intercepting Web Requests
- [ ] Test SQL Injection on login pages to try and bypass them

14. If input appears to be used in a system command, test for command injection

15. If a file upload functionality exists, test for file upload vulnerabilities

16. If the webserver appears to be populating data from a database, test for SQL Injection
- [ ] Test for SQL UNION Injection separately (`test' UNION select 1-- -`)
- [ ] Any input fields, test with sqlmap
- [ ] Test SQL Injection on login pages to bypass them

17. If Accounts, Pages, or other things seem sequential and accessible in GET/POST requests, check for IDOR vulnerabilities

18. If GET parameters seem to be referencing local files, test for file inclusion vulnerabilities / path traversal
- [ ] Run an LFI automation scan against any parameters that reference a file
- [ ] Good candidate example: `http://example.com/index.php?page=goober.html`
- [ ] If LFI exists, look for the following payloads:
	- [ ] `/etc/passwd`
	- [ ] `/var/mail/<username>` (if we can send emails → send and access a PHP webshell)
	- [ ] `/home/<username>/.ssh/id_rsa`
	- [ ] Webserver source code:
		- [ ] `/etc/nginx/nginx.conf`
		- [ ] `/etc/nginx/sites-enabled/default`
		- [ ] `/etc/nginx/sites-available/default`
		- [ ] `/etc/apache2/sites-enabled/default`
		- [ ] `/etc/apache2/sites-available/default`
	- [ ] `/var/log/apache2/access.log` (log poisoning)
	- [ ] `/var/log/nginx/access.log` (log poisoning)
	- [ ] Any file paths from error pages

19. Check user input fields for XSS

20. Check different HTTP verbs to bypass access controls or injection validations

21. If XML input is accepted, test for XML External Entity (XXE) vulnerabilities

22. If the webserver has a web socket or API that is reachable, check if POST/GET requests are injectable with SQLMAP
